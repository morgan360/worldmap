There are many tools that help write, organize and display worlds, but there are limited ways to transfer world data across applications.

OnlyWorlds provides a common standard in world building. This 'blueprint' supports the widest range of worlds and provides data points that can be used in code.

So that you are able to

  • Write content in your favorite writing tool
  • Export to a specialized mapping tool to lay out the world
  • Send to another application to create display pages and share content
  • Load the world into an exploration or simulation game

The framework currently includes

  • Infrastructure for open source development and deployment
  • Infrastructure for hosting and managing worlds
  • A core set of free creation tools
  • The 'blueprint' as a database schema

This database schema defines how a world is broken down to data that includes:

  • 18 element categories
  • Groups and fields to define and integrate them
  • A type and subtype system to classify and organize them
  • Open source, free to use, and community-defined
  • Support any and all third party development, integration, collaboration
  • Users keep complete ownership and control of their worlds

I have been developing world building tools from a game developer perspective since 2011. First in Java, later in Unity, with a highlight in WorldSmith in 2018: An open source world engine where I recreated ancient Sicily of ~500BC with a functional economy simulation and game built on top.
My interests include ((tools for (simulation of)) worlds for) games, and history: political, economy, military, ancient.
Among my favorite technical-creative domains is this breaking down of real world concepts and dynamics into workable data. In my master's thesis I explored the translation of political dynamics into world building structures.

OnlyWorlds intends to provide the definitions and infrastructure for a common file structure that any developer is welcome to integrate or otherwise support in their tooling, for any commercial model.

An open source repository is in the process of being set up, and will be launched together with the framework changes that result from feedback in this closed round.
ETA is late summer 2024. I can give no guarantees on exact data structures, but JSON is likely to remain a primary transfer format.

The current JSON structure includes

  • A header for each category, followed by a list of elements of that category
  • A 'World' header, followed by world metadata
  • 'Progression' and 'Rewards' headers, only relevant to the mobile app at this time

For development reference please see this world file, which additionally has a 'WorldTyping' table that has static data: short descriptors for each Category and, more importantly, the groups-and-fields for each Category.

This file does not have all necessary information. Meta properties of the fields (is it a single-link, multi-link, max-number, ..) I have currently defined in the apps themselves, and this will also be formalized and accessible in the open source release.

I hope that what I do present at this time is enough to theorize or experiment how it might work for you as a developer.
If you have any thoughts or perspectives on what you would want/need in infrastructure, or functionality, or data structure, or anything else, I would be very happy to hear it.
